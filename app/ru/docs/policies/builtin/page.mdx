# Встроенные политики

Goalrail поставляется с набором policies, которые покрывают типовые guardrails.
Их можно включить в session, agent YAML или server config.

Все built-in policies живут под `goalrail.policies.builtins`.

## Safety

- approval перед shell commands;
- approval перед file operations;
- ограничения на network или filesystem access;
- проверки PII и secrets перед model request.

## Cost control

- soft thresholds с approval prompt;
- hard caps для expensive models;
- daily user budget;
- model routing для простых задач.

## Примеры применения

Когда расходы на AI растут, обычно хочется сразу снижать лимиты и добавлять
friction. Более практичный первый шаг - visibility и soft guardrails: команда
видит дорогие sessions, но работа не блокируется преждевременно.

### Сначала сделать расходы видимыми

Начните только с soft thresholds. Инженер видит, что session стала дорогой, и
сам решает, оправдана ли эта стоимость.

```yaml
# Goalrail config (policies block)
policies:
  session_visibility:
    type: function
    handler: goalrail.policies.builtins.cost.cost_budget
    factory_params:
      ask_thresholds_usd: [1.0, 5.0]
      max_cost_usd: 999.0 # effectively no hard cap
  daily_visibility:
    type: function
    handler: goalrail.policies.builtins.cost.user_daily_cost_budget
    factory_params:
      ask_thresholds_usd: [10.0, 25.0]
      max_cost_usd: 999.0
```

Так вы сначала собираете данные о том, где концентрируются расходы, и только
потом решаете, где нужен hard cap.

### Server-wide policy для команды

Практичная стартовая настройка для team deployment: daily visibility на $25,
дополнительный soft check на $50 и hard cap на $100. Модели остаются доступны,
но после $25 в день Goalrail спрашивает подтверждение.

```yaml
# server config (~/.goalrail/config.yaml or /data/config.yaml)
policies:
  daily_budget:
    type: function
    function:
      path: goalrail.policies.builtins.cost.user_daily_cost_budget
      arguments:
        ask_thresholds_usd: [25.0, 50.0]
        max_cost_usd: 100.0
```

Daily cap нужен, чтобы подсветить явно runaway sessions, а не наказать
продуктивную работу.

## Как добавить

Через chat:

```
Ask me before running shell commands.
```

Через YAML:

```
policies:
  shell_approval:
    type: function
    handler: goalrail.policies.builtins.safety.ask_on_os_tools
```

Для своих правил см. [свои политики](/ru/docs/policies/custom).
